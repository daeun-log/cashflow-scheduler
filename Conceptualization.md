# 1. Business purpose
* Project background: 기존의 개인 금융 관리(가계부) 프로그램들을 사용하는데, ott구독료, 교통비 등 매달 정기적으로 발생하는 지출이 매월 1일에 고정되어 있는 것이 아니라서 이를 체계적으로 기록하기 어려웠다. 또, 이 지출들을 포함해서 한 달 소비계획을 세우는 데에도 불편함이 있다.
* Motivation: 자동이체, 구독 시스템 등 날짜 기반의 고정 지출이 증가함에 따라, 다음 주나 다음 달 발생할 수 있는 일시적인 잔액부족 상황을 미리 파악하고 대비할 수 있는 도구의 필요성이 커지고 있다.
* Goal: 사용자의 수입과 지출 예정일을 기반으로 미래의 자금 흐름을 시각화하고, 잔액이 부족해지는 시점을 사전에 경고하는 예측형 금융 스케줄링 시스템 구축하고자 한다.
* Target market: 제한된 수입 내에서 다양한 고정 지출 및 유동 지출을 관리해야 하는 대학생 및 사회 초년생이 타깃 시장이다.

# 4. Concept of operation

1. Join

|||
|:---|:---|
|Purpose|미등록 사용자의 시스템 등록|
|Approach|사용자 정보(ID, PW 등)를 입력받아 중복 여부 확인 후 시스템에 신규 계정으로 저장|
|Dynamics|시스템 기능 사용 전 새로운 사용자가 권한을 부여받고자 할 때 발생|
|Goals|회원가입 기능 구현으로 개인별 금융 데이터를 분리하여 관리할 수 있는 기반을 제공함|

2. Login

|||
|:---|:---|
|Purpose|사용자 인증 및 시스템 접근 권한 획득|
|Approach|ID/PW를 입력하면 저장된 회원정보와 대조하여 로그인 성공/실패 여부를 판단 후 결과 출력|
|Dynamics|기존 사용자가 자신의 금융 데이터를 관리하기 위해 시스템에 접속할 때 발생|
|Goals|로그인 시스템 구현으로 인가된 사용자만 자신의 금융 데이터에 안전하게 접근할 수 있게 함|

3. Logout

|||
|:---|:---|
|Purpose|시스템 접근 권한 해제 및 종료|
|Approach|사용자의 접속 상태를 해제하고 초기(로그인) 화면으로 전환하여 데이터 접근 권한 회수|
|Dynamics|사용자가 시스템 사용을 마치고 데이터를 안전하게 보호하고자 할 때 발생|
|Goals|로그아웃 기능 구현으로 타인의 데이터 열람을 방지하고 접속을 안전하게 종료할 수 있게 함|

4. Add Income

|||
|:---|:---|
|Purpose|신규 수입 내역을 시스템 타임라인에 반영|
|Approach|날짜, 금액, 카테고리 등의 수입 정보를 입력받아 유효성 검사 후 시스템 내역에 추가|
|Dynamics|월급, 용돈 등 새로운 수입이 발생하거나 예정되어 있을 때 발생|
|Goals|수입 내역 추가 기능 구현으로 향후 발생할 자금을 미래 타임라인 연산에 반영할 수 있게 함|

5. Modify Income

|||
|:---|:---|
|Purpose|기존 수입 내역의 오류 수정 또는 상세 정보 변경|
|Approach|수정할 수입 내역을 선택하고 새로운 정보를 입력받아 기존 내역을 덮어쓰기 형태로 변경|
|Dynamics|기존에 등록된 수입 일정이나 금액에 변동이 생겼을 때 발생|
|Goals|수입 내역 수정 기능 구현으로 변동된 수입 정보를 바로잡아 이후 잔액 계산의 정확도를 높이게 함|

6. Delete Income

|||
|:---|:---|
|Purpose|불필요하거나 잘못 등록된 수입 내역 제거|
|Approach|삭제할 수입 내역을 선택하면 재확인 알림을 거친 후 시스템에서 완전히 삭제|
|Dynamics|예정되었던 수입이 취소되거나 잘못 입력된 데이터를 지울 때 발생|
|Goals|수입 내역 삭제 기능 구현으로 취소된 일정을 제거하여 자금 계산의 오차를 방지하게 함|

7. Add Expense

|||
|:---|:---|
|Purpose|신규 단발성 지출 내역을 시스템에 반영|
|Approach|날짜, 금액, 카테고리 등의 단발성 지출 정보를 입력받아 유효성 검사 후 시스템 내역에 추가|
|Dynamics|일회성 소비(쇼핑, 외식 등)가 발생하거나 예정되어 있을 때 발생|
|Goals|단발성 지출 추가 기능 구현으로 예정된 일회성 소비를 자금 흐름에 미리 반영할 수 있게 함|

8. Modify Expense

|||
|:---|:---|
|Purpose|기존 단발성 지출 내역의 상세 정보 갱신|
|Approach|수정할 지출 내역을 선택하고 변경할 정보를 입력받아 기존 데이터를 새로운 정보로 갱신|
|Dynamics|지출 예정 금액이나 결제일이 변경되었을 때 발생|
|Goals|단발성 지출 수정 기능 구현으로 결제일/금액 변동 시 이를 갱신하여 잔액 계산 오류를 방지하게 함|

9. Delete Expense

|||
|:---|:---|
|Purpose|불필요한 지출 내역을 시스템에서 제거|
|Approach|삭제할 지출 내역을 선택하고 재확인 알림을 거친 후 시스템에서 해당 내역을 제거|
|Dynamics|예정된 소비를 취소하거나 잘못된 입력을 지우고자 할 때 발생|
|Goals|단발성 지출 삭제 기능 구현으로 취소된 소비 일정을 제거하여 불필요한 잔액 차감을 막게 함|

10. Search Financial Event

|||
|:---|:---|
|Purpose|조건에 맞는 특정 자금 이동 내역 탐색|
|Approach|검색 조건(키워드, 날짜, 카테고리 등)을 입력받아 조건과 일치하는 내역만 필터링하여 화면에 출력|
|Dynamics|과거 지출 내역을 확인하거나 특정 항목의 지출 액수를 찾고자 할 때 발생|
|Goals|조건별 검색 기능 구현으로 과거 특정 내역이나 카테고리별 지출 규모를 빠르게 찾아볼 수 있게 함|

11. View Projected Balance

|||
|:---|:---|
|Purpose|누적 예상 잔액 및 자금 흐름 시각화|
|Approach|조회 기간을 설정하면 해당 기간의 수입/지출 내역을 날짜순으로 정렬하고, 잔액 증감을 순차적으로 계산하여 타임라인 형태로 출력|
|Dynamics|미래의 특정 시점에 통장 잔고가 얼마일지 종합적으로 파악하고자 할 때 발생|
|Goals|자금 흐름 조회 기능 구현으로 미래의 특정 시점에 통장 잔고가 어떻게 변하는지 시각적으로 파악할 수 있게 함|

12. Simulate Expense

|||
|:---|:---|
|Purpose|가상의 지출이 미래 잔액에 미치는 영향 사전 분석|
|Approach|가상의 지출 금액과 날짜를 입력받아 원본 데이터에 저장하지 않고, 임시로 미래 잔액 변화를 계산하여 결과만 출력|
|Dynamics|큰 지출을 결정하기 전, 해당 소비로 인해 자금 흐름에 문제가 생기지 않는지 테스트할 때 발생|
|Goals|가상 시뮬레이션 기능 구현으로 실제 데이터 조작 없이 큰 지출이 향후 자금에 미칠 영향을 테스트할 수 있게 함|

13. Check Overdraft Alert

|Dynamics|타임라인을 조회하거나 연산할 때 자동으로 백그라운드에서 검사하여 위험이 발견될 때 발생|
|Goals|잔액 부족 알림 기능 구현으로 자금이 마이너스가 되는 위험을 사전에 인지하고 일정을 조정할 수 있게 함|

|Approach|전체 수입/지출 일정을 순차적으로 계산하여 통장 잔고가 최초로 0원 이하가 되는 날짜를 찾아내어 사용자에게 안내|
|Dynamics|사용자가 현재 소비 패턴 유지 시 언제 자금 부족(파산) 상태가 되는지 정확한 시점을 알고자 할 때 발생|
|Goals|자금 고갈일 탐색 기능 구현으로 현재 소비 패턴 유지 시 잔액이 부족해지는 시점을 찾아 사전 대비를 가능하게 함|


# 5. Problem statement

### Overview

* 일회성 수입/지출과 매월 나가는 고정 지출을 하나의 일정으로 합쳐서 날짜별 예상 통장 잔고 제공

* 통장 잔고가 0원 이하로 떨어지는(자금 부족) 날짜를 미리 계산하여 사용자에게 경고 알림 제공

* 실제 지출 전에 가상의 금액을 입력해 보는 시뮬레이션으로 미래의 잔고 변화를 미리 확인

* 로그인 기능으로 개인 금융 정보를 보호하고, 입력한 데이터를 파일로 저장하여 프로그램을 껐다 켜도 기록이 유지되도록 관리

1. Problem #1: 정확한 계산 및 날짜 데이터 정확성 보장
금융 데이터를 다루는 시스템이므로 전산상의 잔액 계산 및 날짜 연산에 어떠한 오차나 누락도 발생해서는 안 된다. 만약 소수점 처리 오류나 월별 일수(28~31일 및 윤년) 계산 오류 등으로 인해 시스템이 잘못된 예상 잔액 리포트를 제공할 경우, 사용자가 이를 믿고 과소비를 하여 실제 카드 대금이 연체되는 등 치명적인 재무적 피해를 입을 수 있다. 이는 시스템의 핵심 존재 목적을 상실하게 만든다.

2. Problem #2: 동적 데이터 누적에 따른 처리 성능 저하 (기술적 문제)
개인의 금융 기록은 과거 내역의 수정이나 미래 일정의 취소, 추가 등 데이터의 삽입과 삭제가 매우 빈번하게 일어난다. 데이터가 수천 건 이상 누적되었을 때 이를 고정된 크기의 단순 배열로 관리하면, 데이터 변동 시마다 전체 요소를 이동시키고 재정렬해야 하므로 극심한 성능 저하가 유발된다. 사용자가 타임라인 조회를 요청할 때마다 시스템이 지연되는 현상을 막기 위해, 데이터 삽입/삭제에 최적화된 자료구조와 빠른 정렬 알고리즘의 적용이 필수적으로 고려해야 한다.

3. Problem #3: 데이터 영속성 부재 및 민감 정보 보호 (보안 및 유지 문제)
사용자가 입력한 수많은 수입/지출 예정 데이터가 프로그램 종료 시 메모리에서 휘발되어 초기화된다면 스케줄러로 사용할 수 없다. 따라서 시스템 종료 시 데이터를 로컬 파일 등에 안전하게 기록하고 불러오는 데이터 영속성이 반드시 구현되어야 한다. 또한, 개인의 자금 흐름은 매우 민감한 사생활 정보이므로 로그인 시스템을 통한 접근 제어를 구현하여, 허락되지 않은 타인이 데이터를 함부로 열람하거나 훼손할 수 없도록 해야 한다.


 # 6. Glossary

| Term (용어) | Description (설명) |
| :--- | :--- |
| Financial Event | 모든 수입이나 지출을 묶어서 부르는 말. 각각 날짜와 금액, 카테고리 정보가 있음. |
| Fixed Expense(고정 지출) | 구독료, 통신비, 교통비 등 매 달 정해진 날짜나 주기에 반복적으로 나가는 예상 지출. |
